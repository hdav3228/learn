Run-prep workflow

The practical order of operations

Dr. Ioerger's instructions and the meeting both emphasize: test small, test local, test on one gene, then scale. The HPRC Faster cluster costs real service units, and a bad 4,018-job array can waste hours of compute budget in minutes.

The safe order
Local validation first, one-gene executable test second, tiny Slurm array third, full array last. This order keeps bugs cheap.
1. Local/Galaxy setup Compile GenomegaMap on Galaxy following the install email. Confirm it produces the porB3 ML output matching Dr. Ioerger's numbers (θ ≈ 0.122, κ ≈ 2.645, ω ≈ 0.899).
2. One-gene tiny test Run the constant model on a single small gene (e.g. porB3 or Rv0001). Check that the output has the expected column format and ~2,000 rows. Compare to the existing porB3 result file.
3. Write the XML generator Use generate_xml.py to loop over gene names, filter FASTA records to good samples, inject paths, and write per-gene XML configs.
4. Small Slurm test Submit 3-5 genes as a Slurm array on Faster. Inspect logs, output files, runtime, and memory usage.
5. Scale toward ~4,018 genes Only after the small array produces correct outputs, submit the full gene set for each cohort.

Pre-scale checklist

  1. Run one dry validation. Confirm Rv0001 keeps 178 diabetes and 175 non-diabetes records.
  2. Generate one real config pair. Create one filtered FASTA and XML for a single gene/cohort so you can inspect the exact paths.
  3. Trace the path assumptions. Decide whether the XML will be run from cleaned-genes, copied to Faster scratch, or rewritten with absolute paths.
  4. Prepare a tiny job list. Pick 3-5 genes for the first array test; do not use all 4,018 as the first Slurm attempt.
  5. Write down expected output shape. The trace table should include iteration, loglikelihood, theta, kappa, and omega0.
Cluster caution: service units
Never submit thousands of jobs until one job and one mini-array have produced the exact output table you expect. The meeting explicitly warned that a bad array wastes service units. Prefer conservative resource requests: short wall times, small memory, and one CPU per job.
Workflow vocabulary
SSH means logging into the cluster from your terminal. Module/environment setup means loading the software GenomegaMap needs. Slurm is the cluster's job scheduler. A job array is one submission that runs the same script many times, usually once per gene. Service units are the compute budget you spend when jobs run.

XML generator

What the XML file controls

GenomegaMap does not accept simple command-line flags for inputs and outputs. You must feed it an XML configuration file that specifies everything: the input FASTA path (the <seqs> block), the output destination, the MCMC hyper-parameters (iterations, thinning interval), and prior distributions.

Think of XML as a config object

The template is the class definition. Each generated XML file is one instance with a specific gene, cohort, input FASTA path, and output trace path.

The dangerous fields are paths

The statistical settings may stay fixed, but the input and output paths must change for every gene. If they do not, the array silently becomes wrong even if jobs appear to run.

Open XML field reference and generator mechanicsReference

What each XML field does

<seqs> (input sequences)
Points to the aligned FASTA file for one gene. This is the field that must change for every gene. If all 4,018 jobs read the same <seqs> tag, they will all analyze the same gene.
Output file path
Where GenomegaMap writes the MCMC trace table. This must also be unique per gene; otherwise 4,018 parallel jobs overwrite the same file simultaneously.
Iterations
How many MCMC steps to run. Usually set high: 10,000 for a test, 100,000 or 1,000,000 for a real run.
Thinning interval
Save every nth iteration. If you run 10,000 iterations and thin by 5, you get 2,000 saved rows. This prevents gigabyte output files filled with highly-correlated adjacent samples.
Prior distributions
The XML can specify prior distributions (e.g. Gamma priors for variance parameters) and starting values. Do not modify these unless Dr. Ioerger explicitly says to.
Model type
Constant, independent, or window. You are running constant first.

5. Bayesian foundations

The Jun Liu & Lawrence paper

You are required to read the first half of the Jun Liu & Lawrence (1999) paper on Bayesian inference in biopolymer models. It strips away the biology and explains probability from the ground up. The reading guide is at references/instructions/bayesian_inference_models.md and the full text is at references/extracted-text/bayesian_inference_models.txt.

If you only remember one sentence
Bayesian inference takes your prior beliefs about a parameter, combines them with observed data through a likelihood function, and produces a posterior distribution that quantifies both the best estimate and the uncertainty.

Know this for Dr. Ioerger

  • What a likelihood is and why GenomegaMap works in log-likelihood space.
  • Why the posterior is a distribution, not a single point estimate.
  • Why conjugate priors are convenient in toy examples but not enough for GenomegaMap.
  • Why MCMC samples are a practical approximation to an intractable posterior.

Do not overinvest yet

  • Full derivations of every biopolymer model in the Liu paper.
  • Every prior family in the XML schema.
  • Change-point/window math before the constant model works.
  • Biological interpretation of every gene before the scan finishes.

The Bayesian Decoder

posterior ∝ likelihood × prior
Parameter
An unknown number the model is trying to estimate. GenomegaMap estimates ω, κ, and θ jointly.
Likelihood Function
Connects observed data to parameters. It answers: if the true ω was 1.5, how probable would it be to see this exact pattern of synonymous and nonsynonymous mutations? Higher likelihood means that parameter value explains the data better.
Prior Distribution
Your statistical belief about the parameter before looking at the data. Usually modeled as a broad distribution so the data can dominate. The XML file specifies these.
Joint PDF
The product of the likelihood and the prior. In practice the denominator in Bayes' theorem (the marginal likelihood, integrating out all parameters) is often dropped, making the posterior proportional to the joint PDF.
Posterior Distribution
The updated distribution over parameter values after combining prior and likelihood. This is the answer GenomegaMap outputs: not a single number, but a spread.
Posterior Mean
The average of the sampled posterior values. Useful as a summary, but not enough by itself. A mean of 1.2 from a very wide distribution is different from a mean of 1.2 from a narrow one.
Credible Interval
A range containing most of the posterior probability, often 95%. If the 95% credible interval for ω crosses 1, the evidence for positive selection is weaker even if the mean is above 1.
Conjugate Priors
A mathematical "hack" to make the equations solvable in closed form. If you pick the right prior distribution family for your likelihood, they multiply together to produce a posterior of the same distribution family, avoiding complex integrals.
Log-likelihood
The natural log of the likelihood. Because likelihood products across thousands of data points get astronomically small, computations are done in log space to avoid numerical underflow.

Conjugacy explained

θ

Binomial + Beta

For binary events (like a coin flip), a Binomial likelihood combined with a Beta prior yields a Beta posterior. The new Beta parameters are just the old ones plus the observed counts.

μ

Normal + Normal

If your likelihood is Normal with known variance, a Normal prior over the mean yields a Normal posterior. The posterior mean is a weighted average of the prior mean and the data mean.

σ²

Normal + Inverse-Gamma

If estimating variance, combining a Normal likelihood with an Inverse-Gamma prior yields an Inverse-Gamma posterior. This is relevant for GenomegaMap's variance priors specified in the XML.

The Coin Flip Example (from the meeting)

1
Setup. You flip a coin 30 times and get 12 heads and 18 tails. You want to estimate the true weight θ (probability of heads).
2
Likelihood. Binomial distribution: the probability of seeing exactly 12 heads in 30 flips as a function of θ.
3
Prior. A Beta(α, β) distribution. If you know nothing, use α = β = 1 (uniform). If you suspect a fair coin, use α = β = 10 (peaked at 0.5).
4
Posterior. Because Beta is conjugate to Binomial, the posterior is just another Beta: Beta(α + 12, β + 18). No integration needed.
5
Result. The posterior distribution peaks near 40% (12/30), but the full distribution explicitly shows you the probability that the coin could be 30% or 50% weighted. This uncertainty is exactly what GenomegaMap gives you for ω — not just a point estimate, but a spread.

Why GenomegaMap gives a distribution, not a number

1
Data: many TB isolates share the same gene, with occasional mutations.
2
Question: what ω value would make this pattern of synonymous and nonsynonymous mutations plausible?
3
Output: not one answer, but a distribution of possible ω values. If the distribution is mostly below 1, think purifying selection. If it is mostly above 1, think positive selection. If it is wide and crosses 1, the data is too weak to decide.
Best reading order
Start with the reading guide at references/instructions/bayesian_inference_models.md. Focus on the coin-flip example. Then read the first half of the extracted text. Then come back to this page and make sure you can explain what a prior, likelihood, and posterior are without looking.

6. Gibbs vs Metropolis-Hastings

When closed-form solutions fail

Conjugate priors only work for simple models. For GenomegaMap's Dirichlet-multinomial codon model with three parameters (ω, κ, θ), you cannot analytically calculate the posterior distribution. You must sample from it to build an empirical histogram.

Gibbs Sampling

Used when you can't calculate the joint posterior for all parameters at once, but you CAN calculate the conditional posterior of one parameter when the others are held fixed.

Metropolis-Hastings

A "knowledge-free" fallback for when even conditional posteriors are too hard. You propose random moves and probabilistically accept or reject them based on likelihood ratios.

How Gibbs Sampling works (3 parameters: θ, κ, ω)

1
Start: Guess initial random values for θ₀, κ₀, and ω₀.
2
Update θ: Fix κ and ω at their current values. Mathematically derive the conditional posterior distribution of θ given κ and ω. Draw a random value from that distribution. This is θ₁.
3
Update κ: Fix θ₁ (updated) and ω₀ (not yet updated). Draw a new κ value from its conditional posterior. This is κ₁.
4
Update ω: Fix θ₁ and κ₁. Draw a new ω value from its conditional posterior. This is ω₁.
5
Record: Save (θ₁, κ₁, ω₁) as one MCMC sample — one row in the output file.
6
Repeat: Go back to step 2 thousands of times. The sequence of parameter triplets "wanders" around parameter space, staying near the true peak. After a burn-in period, the histogram of these samples is the approximated posterior.

How Metropolis-Hastings works

1
Current state: You are at parameter values (θ, κ, ω) with a known log-likelihood.
2
Propose: Randomly perturb one parameter. For example, change ω from 0.90 to 0.95 by adding a small random number drawn from a proposal distribution (often a normal distribution centered at 0).
3
Score: Calculate the log-likelihood at the new parameter values. Compare it to the old log-likelihood.
4
Accept or reject: If the new log-likelihood is higher, always accept. If it is lower, accept with probability proportional to the ratio exp(new_loglik − old_loglik). This means worse proposals are less likely to be accepted, but they sometimes are, allowing the chain to escape local minima.
5
If rejected: The chain stays at the old values. In the output file, this appears as a repeated row — the same parameter values on consecutive lines. This is normal and expected.
Why repeated rows happen in MCMC output
A proposed move was rejected, so the chain kept its old parameter values. This is not a bug. If you see many consecutive identical rows, the chain may be stuck in a mode or the proposal distribution is too wide. If you rarely see repeats, proposals may be too small. Both are diagnostic signals.
Simulated Annealing analogy
The acceptance ratio in Metropolis-Hastings is conceptually similar to Simulated Annealing: worse proposals can be accepted with a probability like e−Δ/T. The difference is MCMC does not reduce the "temperature" over time — it maintains a constant acceptance policy to correctly sample the posterior.

MCMC vocabulary

Burn-in
Early MCMC samples discarded because the chain may not yet represent the target posterior. The first few hundred iterations often drift as the chain finds the right region of parameter space.
Thinning
Saving every nth sample. Adjacent MCMC samples are correlated; thinning reduces that autocorrelation and keeps file sizes manageable.
Convergence
The chain has "settled in" and is sampling from the true posterior. You can check convergence by plotting the trace of ω over iterations — it should look like random noise around a stable mean, not a trend.
Trace plot
A plot of a parameter value (y-axis) vs iteration number (x-axis). Used to visually assess convergence and diagnose problems.
Mixing
How well the chain explores the parameter space. Good mixing = the trace plot looks like "hairy caterpillar" noise. Bad mixing = the trace plot has long flat stretches or sudden jumps.
Acceptance rate
The fraction of proposed moves that are accepted. Too high (>0.5) means proposals are too timid. Too low (<0.1) means proposals are too bold. Typical target: 20-40%.
Test sampling yourself
Install R locally or use Galaxy. Run rnorm(1000, mean=0, sd=1) to draw 1,000 random values from a normal distribution. Plot a histogram. This is exactly what MCMC does: draw samples, build a histogram, and treat the histogram as the posterior.

Reading the porB3 output step by step

1
Burn-in check: The first few rows show large log-likelihood changes (−1949 → −1934). The chain is still settling. Discard the first ~200 iterations as burn-in.
2
Notice repeated rows: Lines 11 and 12 are identical (iteration 55 and 60 have the same loglikelihood/theta/kappa/omega). This means the proposed MCMC move at step 55 was rejected. Normal behavior.
3
ML comparison: The install email shows maximum likelihood estimates: θ ≈ 0.122, κ ≈ 2.645, ω ≈ 0.899. The MCMC samples should cluster around these values, but with spread representing uncertainty.
4
Interpret ω: Most sampled ω values in the output are near 0.8-1.0. This is roughly neutral or very mildly purifying — the gene is not under strong positive selection.

Recognize problems before they become results

If you see: loglikelihood keeps trending down/up for 1000+ iterations
→
Check: the chain hasn't converged. Run more iterations or check input data.
If you see: omega0 is exactly the same for 100+ consecutive rows
→
Check: the chain is stuck. The proposal step size may be too large, causing all moves to be rejected.
If you see: all genes have identical output files
→
Check: the Slurm array index is not selecting different genes, or the XML generator used the same input path for all configs.
If you see: theta or kappa values that are wildly different from the ML estimates
→
Check: the input file may be malformed, or the XML may have wrong priors.

8. Change-point models

The coin-swap analogy from the meeting

Dr. Ioerger described a scenario where someone flips coin 1 for a while, then secretly swaps it for coin 2 mid-sequence. You must estimate the weights of both coins AND the exact step where the swap occurred. This is a direct analogy to GenomegaMap's sliding-window model.

The coin game
analogy

You observe a sequence of 100 flips: the first 60 are mostly heads, the last 40 are mostly tails. There are three unknowns:

  • θ₁ — the weight of coin 1.
  • θ₂ — the weight of coin 2.
  • A — the changepoint (the flip number where the swap happened).

The Bayesian approach treats all three as random variables, samples them jointly via MCMC, and returns posterior distributions for all three. The posterior for A will peak near the true swap position.

In GenomegaMap
biology

Replace "coins" with "regions of a gene" and "weights" with "ω values":

  • The first 200 codons might have ω = 0.3 (strongly purifying).
  • The next 100 codons might have ω = 2.5 (strong positive selection).
  • The last 200 codons might return to ω = 0.4.

The independent and window models try to find these boundaries. The constant model (your current assignment) averages them all into one ω, which is why it's simpler but less informative.

Bayesian sequence segmentation
The Jun Liu paper describes a Bayesian sequence segmentation algorithm that is conceptually similar to how GenomegaMap's sliding-window model works: it tries to find regions of the genome with different underlying properties. Understanding the coin-swap example prepares you for the more complex models later this summer.

What you should be able to say out loud

GenomegaMap uses a Bayesian framework with MCMC sampling to estimate the posterior distribution of ω for each TB gene. The XML configuration file specifies inputs, outputs, and prior distributions. We run the constant model first because it gives one gene-wide ω estimate, which is the cleanest first pass across ~4,018 genes. Each gene needs its own XML config with unique file paths so the Slurm array jobs don't overwrite each other. The MCMC output is a tab-separated trace file where each row is a saved sample of (θ, κ, ω). After discarding burn-in, the histogram of the ω column is the empirical posterior. We compare diabetes vs non-diabetes posteriors gene by gene to find candidates for condition-specific selection.

Mini glossary for this meeting

XML configuration
The input file format GenomegaMap uses instead of command-line flags. Controls model type, input/output paths, iterations, thinning, and priors.
gcat
The gcat-project runtime that GenomegaMap depends on. The actual executable is ~/gcat-project/gcat. You pass it an XML file as the only argument.
Conjugate prior
A prior distribution family that, when multiplied by the likelihood, produces a posterior of the same family. Simplifies math but only works for certain model/distribution pairings.
Proposal distribution
In Metropolis-Hastings, the probability distribution used to generate candidate moves. Typically a normal distribution centered at the current parameter value.
Acceptance ratio
The probability of accepting a proposed MCMC move. Equals exp(new_loglik − old_loglik) when the proposal is symmetric. Better moves always accepted; worse moves sometimes accepted.
Markov chain
A sequence where each state depends only on the previous state, not the entire history. MCMC chains have this property: the next parameter set depends only on the current one.
Monte Carlo
Using random sampling to approximate a quantity. The "Monte Carlo" in MCMC means we are using random draws to approximate the posterior distribution.
Service units
HPC compute allocation currency. Each CPU-hour on Faster costs service units. A bad 4,018-job array with a 1-hour walltime could burn 4,018 SU in one submission.
