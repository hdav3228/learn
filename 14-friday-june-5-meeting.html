Faster Mental Model

How to think about HPRC

Faster has login nodes, compute nodes, home storage, and scratch storage. The cluster may feel slow even when you are not computing because shared storage can be hammered by other users' jobs.

ComponentUse it forWatch out for
Login node Shell work, editing scripts, checking files, small Python preprocessing. Do not run large GenomegaMap workloads here.
Compute node Actual GenomegaMap runs submitted through Slurm. You request cores, memory, walltime, and possibly exclusive access.
Home directory Small scripts, important notes, and files worth backing up. Quota is small. Do not unpack large ORF/result folders here long term.
Scratch Large data, temporary ORFs, batch outputs, and generated run folders. No backup guarantee. Keep crucial scripts/results copied elsewhere.
Slurm queue Scheduling jobs onto available compute resources. Scheduling is not pure first-come first-served; walltime/resource requests matter.

Easy-to-miss Slurm failure
If sbatch says a directive is invalid, check for copied non-breaking spaces or odd dash characters in the #SBATCH lines. Re-typing the directives directly in the terminal editor fixed that issue during the meeting.

Script Template

The shape of the 10-gene job

The exact paths will depend on where GenomegaMap, gcat, XML files, and FASTA files live on Faster. The stable pattern is: define Slurm resources first, load environment paths, cd to the run directory, start one backgrounded gcat process per gene, then wait.

10-gene constant-model skeleton
adjust paths 
#!/usr/bin/env bash
#SBATCH --job-name=genomegamap_constant_test
#SBATCH --nodes=1
#SBATCH --ntasks-per-node=10
#SBATCH --cpus-per-task=1
#SBATCH --mem=25G
#SBATCH --time=01:00:00
#SBATCH --output=slurm-%j.out
#SBATCH --mail-type=BEGIN,END,FAIL
#SBATCH --mail-user=<your-email>@tamu.edu

set -euo pipefail

# Adjust these to the actual Faster install paths.
export LD_LIBRARY_PATH="$HOME/software/gcat-project/lib:$HOME/software/genomegamap/build/genomegaMap/lib:${LD_LIBRARY_PATH:-}"

# Make relative XML/input/output paths unambiguous.
cd "$HOME/JaegerTest"

genes=(Rv0001 Rv0002 Rv0003 Rv0004 Rv0005 Rv0006 Rv0007 Rv0008c Rv0009 Rv0010c)

for gene in "${genes[@]}"; do
  "$HOME/software/gcat-project/gcat" "xml/diabetes/genomegamap.constant.${gene}.xml" &
done

wait
LineMeeting ruleReason
#SBATCH block All Slurm directives must come before ordinary shell commands. Slurm parses these before running the script.
--ntasks-per-node=10 Use 10 for the small test. One GenomegaMap process uses one core, so 10 genes use roughly 10 cores.
--mem=25G Start small for 10 jobs. The full-node example used much larger memory, but the 10-gene smoke test does not need that.
LD_LIBRARY_PATH Point to gcat/GenomegaMap dynamic libraries. The first run failed until the correct nested GenomegaMap library path was included.
& and wait Background each gene run, then wait for all of them. This creates parallel work inside one Slurm job without letting the script exit early.
Conda note
Conda may auto-activate in an interactive shell, but Slurm jobs do not always run the same shell startup files. If the executable or libraries are missing inside Slurm, explicitly source the right conda activation script in the Slurm file before running gcat.

Files And Paths

Transfer fewer files on purpose

The meeting exposed how painful repeated scp can be with Duo prompts. When moving multiple XML or FASTA files, tar them on the source machine, copy one tar file, then untar on Faster.

Path ruleWhy it matteredAction
Keep test files close together. Deep mixed paths made the first Slurm test harder to debug. For the 10-gene test, keep XML, FASTA, and output directories simple and predictable.
Create output directories before running. The first successful-looking run did not write expected outputs until output folders existed. Use mkdir -p outputs/diabetes or update XML output paths to an existing directory.
Watch Rv0008c and Rv0010c. Some early gene names include the c suffix. Use exact filenames from the ORF folder or generated gene list, not guessed names.
Delete large unpacked test tarballs when done. Home quota is small and ORF archives expand quickly. Use du -sh . to confirm what is consuming space.

Scaling Strategy

How to avoid wasting service units

GenomegaMap is not internally parallel for one gene, but the project is parallel across genes. The hard part is keeping cores busy without launching a fragile, disk-heavy, hard-to-debug run.

Scaling issueMeeting guidanceProject implication
Diabetes vs non-diabetes Submit separate jobs for the two cohorts. Separate outputs, logs, and summary tables stay easier to audit.
Exclusive nodes Use only when you can keep most cores busy. Do not request exclusive access for 10 jobs; consider it for production batches near 60+ tasks.
Task count One gene run uses one core, but Linux can run more processes than cores via time slicing. You can submit hundreds in one script, but test carefully before starting very large batches.
Stragglers Runtime scales mainly with gene length, so long genes can keep a node busy after short genes finish. Estimate runtime from timestamps and consider batching by expected gene length.
Disk pressure Many jobs starting together all read inputs and write outputs at once. Starting 1,000 processes at once may stress shared storage even if CPU scheduling works.
Walltime Build in buffer, but do not always request the maximum. The queue may schedule shorter jobs sooner; jobs are killed if the time limit is reached.

Load-balancing logic

1
Start with 10 genes. This proves Slurm, paths, libraries, output folders, and basic runtime.
2
Use timestamps. After the run, compare output file modification times to estimate per-gene durations.
3
Relate duration to gene length. Plot runtime versus sequence length so production walltime is based on evidence.
4
Scale in cohorts. Run diabetes and non-diabetes as separate batches once the small run and summary script are stable.

MCMC Output

How to read the trace files

The meeting tied Slurm execution back to statistics. A GenomegaMap output file is an MCMC trace: early samples may be burn-in, saved rows may be thinned, and omega0 should be summarized as a posterior distribution.

TopicMeeting noteUse in analysis
Constant iterations The constant-model default is 10,000 iterations. Use 10,000 for the next 10-gene test unless changed.
Window iterations The window-model default is much larger, around 1,000,000; prior work may have used fewer. Do not switch to sliding/window until the constant model works.
Burn-in Discard early samples while the chain moves from random initialization toward equilibrium. For 10,000 iterations, 2,000 is a reasonable 20% test burn-in.
XML vs post hoc burn-in You can set burn-in in XML, or keep all rows and discard during post-processing. Keeping all rows preserves trace diagnostics; post-processing can enforce the final rule.
Thinning A thinning value of 5 saves every fifth sample. Remember this when translating iteration count into saved row count.
Diagnostics Line plots, heat maps, autocorrelation, and replicate chains help check convergence. At minimum, plot omega0 over iterations for the 10-gene test.
Credible intervals
For each gene, sort post-burn-in omega0 samples and report a credible interval plus a mean/median. Conventionally, a 95% credible interval uses the 2.5% and 97.5% quantiles; if Dr. Ioerger wants a different middle interval, record that rule explicitly.

Interpretation target

Values below 1 suggest purifying selection, values near 1 suggest neutral evolution, and values above 1 suggest positive selection. Dr. Ioerger expects the first 10 genes to be unremarkable, so they are mainly an engineering and diagnostics test rather than a biological discovery set.

Ad-Hoc pN/pS Challenge

New task from Dr. Ioerger

Build a lightweight independent calculation from the sequence files: for every gene, count synonymous and non-synonymous mutations in the diabetes cohort and repeat the same calculation in the non-diabetes cohort. This gives a quick sanity-check ranking that can be compared to the more rigorous GenomegaMap output.

What the first table should answer

The table should make it easy to rank genes by NS-over-S enrichment in diabetes and by the biggest DB-vs-NDB difference in a pN/pS-like score.

Per gene S + NS Count synonymous and non-synonymous observed changes.
Cohorts DB / NDB Run the tally independently for diabetes and non-diabetes samples.
Rank 1 DB NS/S Find genes with the strongest NS-over-S enrichment in diabetes.
Rank 2 ΔpN/pS Find genes with the largest cohort difference.
PieceFirst implementation ruleWhy it matters
Inputs Use the per-gene sequence files for DB and NDB after the same good-sample filtering rule. The ad-hoc method should compare the same cohorts as GenomegaMap.
Reference codon Compare each sample codon to the reference/aligned gene codon in frame. Synonymous vs non-synonymous classification only makes sense at codon boundaries.
Classification Translate reference and alternate codons; same amino acid means S, different amino acid means NS. This is the biology core of the script and should be easy to unit test.
Ambiguous sites Skip gaps, partial codons, stop-codon surprises, and ambiguous bases in the first pass. A clean first table is better than quietly counting uncertain calls.
Counting rule Record both unique mutation counts and per-isolate counts if possible; pick one as the displayed ranking rule. Unique events and repeated observations answer slightly different questions.
Output columns Gene, cohort, sample count, S count, NS count, NS/S score, skipped codons, and notes. These fields make later ranking and QC transparent.

Ranking formulas to keep explicit

1
DB enrichment. Rank genes by (NS_DB + 0.5) / (S_DB + 0.5) so zero-count genes do not explode.
2
Cohort difference. Compare log2((NS_DB + 0.5)/(S_DB + 0.5)) against the same score in NDB.
3
Cross-method check. After GenomegaMap finishes, compare the top ad-hoc genes with the top omega0 or DB-vs-NDB omega-difference genes.
How to present it
Call this an independent ad-hoc sanity check, not a replacement for GenomegaMap. The value is speed and interpretability: if both methods point to similar genes, those hits become easier to prioritize for deeper review.
