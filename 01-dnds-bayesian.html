GenomegaMap: Bayesian dN/dS at genome scale

A pathogen evolving inside a host has skin in the game; the random mutations its DNA picks up are subjected to natural selection in real time. The dN/dS ratio is how we read those tea leaves. GenomegaMap is the modern, scalable, fully Bayesian way to estimate it across whole genomes from tens of thousands of clinical isolates.

What dN/dS is See MCMC live Three model variants
In plain English Imagine watching a pathogen mutate over thousands of patient infections. Some mutations change the protein it makes; some don't. If selection is doing nothing, both kinds happen about equally (per opportunity). If selection is helping certain protein changes win — say, mutations that escape an antibiotic — then protein‑changing mutations pile up faster. The ratio of "protein‑changing" to "silent" mutations is dN/dS, also called ω (omega). ω > 1 = protein is being adaptively reshaped. ω < 1 = protein is locked in place by selection. GenomegaMap is just the Bayesian recipe for estimating ω rigorously from real data.

1. What dN/dS measures

Why this section matters

Before we can infer selection, we need a number that quantifies it. dN/dS (ω) is that number. This section defines it and explains why "just count mutations" doesn't work as an estimator.

Every protein‑coding nucleotide change is one of two flavors: it either swaps the encoded amino acid (nonsynonymous, "dN" type) or it doesn't (synonymous, "dS" type). The genetic code is degenerate — most amino acids are coded by multiple codons — so synonymous changes happen all the time without any direct fitness consequence. They're a decent proxy for the raw mutation rate.

Nonsynonymous changes do alter the protein. If selection is purifying (the wild‑type protein is already fit), most of these get filtered out. If selection is positive (some variant protein is fitter than wild type), nonsynonymous changes accumulate faster than they would by chance. So:

ω = dN / dS
ω < 1
Purifying selection. Most amino‑acid‑changing mutations are deleterious and removed. Typical of essential housekeeping genes. Most genes in any genome look like this.
ω ≈ 1
Neutral evolution. Either the protein is unconstrained, or selective pressure is too weak to detect in this dataset.
ω > 1
Positive (diversifying) selection. Amino‑acid changes are accumulating faster than synonymous ones. This is the signature of an adaptive arms race — drug resistance, immune evasion, host adaptation.
Important nuance
ω is a per‑gene (or per‑codon, per‑window) quantity. A single ω > 1 codon embedded in a constrained gene with overall ω = 0.3 is invisible at gene level. Where you average matters enormously — that's the entire point of the Ioerger lab's TB scans.

Why it's not just N/S

If you naively count up nonsynonymous and synonymous mutations and divide, you get a biased estimate. The reason is that there are more possible nonsynonymous than synonymous changes per codon. If you mutate one base in a random codon, ~75% of single‑nucleotide changes alter the amino acid — so you'd expect 3× more nonsynonymous than synonymous mutations even if all mutations were neutral. dN and dS are normalized rates per nonsynonymous and synonymous site respectively; ω compares them on equal footing.

2. The codon substitution model (Nielsen & Yang 1998)

Why this section matters

Just counting mutations gives biased answers (you saw why in §1). Instead, we write down a probability model of how codons mutate over time that has ω built in as a tunable knob. Then we can do statistics on real data. This section is the model.

In plain English A "rate matrix" Q is just a 61×61 lookup table: row i, column j says "rate at which codon i turns into codon j." Most entries are 0 (because two codons rarely differ at >1 base in one step). The rest are simple formulas in four parameters: μ (overall mutation speed), κ (transitions are mechanistically faster than transversions), πj (target codon frequency), and the star of the show ω (multiplier on amino‑acid‑changing transitions/transversions).

To estimate ω rigorously we need a probabilistic model for codon evolution. The standard workhorse is NY98: a 61×61 (60×60 if we drop stop codons) instantaneous rate matrix Q where Qij is the rate at which codon i mutates to codon j. The trick is to parameterize Q with a small number of biologically meaningful parameters so we can do statistics:

Qij = {0      if codons differ at >1 position
 μ πj      synonymous transversion
 μ κ πj    synonymous transition
 μ ω πj    nonsynonymous transversion
 μ ω κ πj   nonsynonymous transition}
μ
Neutral mutation rate scaling factor.
κ
Transition / transversion rate ratio. Empirically >1 (transitions, A↔G or C↔T, are mechanistically more frequent than transversions). Values of 2–4 are typical for mycobacterial genomes.
ω
The dN/dS we want — a multiplier on nonsynonymous rates.
πj
Equilibrium frequency of target codon j. Reflects codon‑usage bias.

The model assumes only single‑nucleotide changes per "instant" of evolutionary time (multi‑nucleotide events are rare on the timescales we care about). Notice that ω only enters the rates that change the amino acid; it's literally a knob on dN.

3. The naive estimator (and why we need better)

Why this section matters

Before reaching for full Bayesian machinery, it's worth seeing what the simplest possible recipe looks like — both because the lab uses it as a sanity check, and because seeing it fail tells us why we need GenomegaMap.

The Shatby thesis runs a "manual" dN/dS as a sanity check before turning to GenomegaMap. The recipe is:

  1. Count unique nonsynonymous SNPs N and unique synonymous SNPs S across all isolates.
  2. Add pseudocounts to handle low‑count genes: 2 to N, 1 to S (the 2:1 pseudocount roughly matches the 2:1 expected ratio under neutrality, so the prior is "neutral").
  3. Compute a log‑fold change:
    LFC = log2((N + 2) / (S + 1))
  4. Optionally, χ² test against the expected 2:1 split.
Why the manual approach is not enough
It's quick, but it has serious problems:
  • No notion of uncertainty. You get a point estimate, not a credible interval.
  • Doesn't account for transitions vs. transversions or codon‑usage bias.
  • Doesn't scale ω against the opportunity for nonsynonymous changes.
  • Lineage‑specific mutations get awkwardly excluded by ad‑hoc rules.
GenomegaMap is the principled fix.

3

4
Sanity check. χ² ≈ 7 against the 2:1 expectation, p ≈ 0.008. So the result is unlikely under neutrality — but we have no uncertainty interval, no proper handling of κ/codon usage, no idea if this is one hot codon or a gene‑wide signal. This is exactly the gap GenomegaMap fills.

4. Going Bayesian

Why this section matters

A point estimate of ω hides uncertainty. Genes with few mutations look like they have wildly extreme ω (we saw this above). Bayesian inference replaces "best guess" with "full distribution of plausible values" — which is what GenomegaMap reports.

In plain English A point estimate says "ω is best estimated at 1.7." A posterior says, "conditional on this model, prior, and data, here is the full distribution of plausible ω values." For genes with little information, that distribution can remain wide. The width is part of the result, not an inconvenience to hide.

Bayes' rule as a lab habit

1
Prior: before looking at this gene, the model allows many possible ω values, but it does not assume every extreme value is equally likely.
2
Likelihood: for a proposed ω, ask whether the observed synonymous and nonsynonymous codon changes would make sense under that value.
3
Posterior: combine those two pieces. The result is a curve over possible ω values. That curve is what the MCMC samples are trying to represent.

We want, for each gene, the full posterior distribution on ω given the observed allele counts x. By Bayes' rule:

p(ω, κ, μ | x) ∝ p(x | ω, κ, μ) · p(ω) p(κ) p(μ)

The left‑hand side is what we report (mean, median, 95% credible interval). The right‑hand side is just the likelihood times the prior. The likelihood is determined by the codon model. Priors used in Wilson's 10,209-genome application were calibrated from a preliminary hierarchical analysis and then widened to avoid being overly informative:

Why credible intervals instead of p‑values? The Bayesian credible interval has a clean interpretation: "given the model and prior, there is a 95% probability that ω lies in this range." The Shatby thesis flags a gene as significantly under positive selection when the entire 95% CI sits above 1 — i.e., the posterior puts 97.5% of its mass on ω > 1.

The likelihood: Multinomial‑Dirichlet

At each codon site, given that the "ancestral" codon was A, the predicted distribution over observed allele counts x follows a Multinomial‑Dirichlet form. Wilson 2020 derives this by combining a Poisson‑Random Field model of allele frequencies with the codon rate matrix:

Pr(x | A) = ( n⁄x ) · B(x + αA) / B(αA)

where n is the sample size, αA is a 61‑vector of Dirichlet concentration parameters derived from Q, and B is the multivariate beta function. The ancestral codon is unknown so we marginalize:

Pr(x) = ΣA πA · Pr(x | A)

This is one site's contribution to the likelihood. The total log‑likelihood for a gene is the sum over its codon sites — under the assumption that sites are independent. We'll return to that assumption.

Bayesian vs. frequentist takes
Classical PAML‑style methods optimize a likelihood over a fixed phylogeny and report a maximum‑ likelihood ω plus a likelihood‑ratio test. GenomegaMap instead samples the posterior, so the uncertainty about ω is built into the answer rather than computed from a Hessian.

5. Metropolis‑Hastings MCMC

Why this section matters

We just wrote down the posterior we want, but it is intractable to normalize and summarize directly. MCMC builds an approximate sample from that posterior using repeated stochastic updates. It is the inference engine inside GenomegaMap.

In plain English Imagine the posterior is a hilly landscape, and you can only feel the height under your feet (not see the whole map). MCMC is a procedure where you take little random steps, but you're more likely to step uphill than down. Run it long enough and the fraction of time your feet spent in each region matches the height of the landscape there. That's it. The fancy formulas just spell out exactly how to bias the steps so this works.

Computing the posterior in closed form is hopeless — there are 60 codons, three continuous parameters, and a likelihood that's a product over thousands of sites. But we can sample from it. Markov Chain Monte Carlo (MCMC) is designed to build a Markov chain whose stationary distribution is the posterior we want. In practice, finite chains can mix slowly or fail to explore important regions, so diagnostics are part of the analysis.

Output-file decoder
A GenomegaMap output table is not one row per gene result. It is usually one row per saved MCMC sample. If you see columns like iteration, likelihood, ω, κ, and θ, each row is one visited point in parameter space. The final gene summary comes later, when you take means, credible intervals, and diagnostic plots from those rows.

The Metropolis‑Hastings (MH) update is one of the simplest and most general MCMC moves:

  1. Start at a random point θ0 in parameter space.
  2. Propose a new value θ' drawn from a proposal distribution q(θ' | θt) — typically a small Gaussian random walk in log space.
  3. Compute the acceptance ratio
    α = min(1, [p(θ' | x) · q(θt | θ')] / [p(θt | x) · q(θ' | θt)])
  4. With probability α, accept (θt+1 = θ'). Otherwise stay (θt+1 = θt).
  5. Repeat. Use the configured burn-in period, then diagnose the retained chain.

The useful part of step 3 is that the intractable posterior normalization constant cancels in the ratio. We only need to evaluate the unnormalized density (likelihood × prior). If a proposal is symmetric, its q terms also cancel; for asymmetric proposals, the Hastings correction must remain.

Acceptance is a diagnostic, not a pass/fail number
Very high acceptance can mean proposals are too small, while very low acceptance can mean they are too aggressive. There is no single ideal rate for every proposal and target distribution; inspect acceptance together with trace behavior, autocorrelation, and independent chains.

What about reversible‑jump MCMC?

Standard MH assumes the parameter space is fixed. But GenomegaMap's sliding window variant has a variable number of windows — splitting and merging changes the dimension of the parameter vector. Reversible‑jump MCMC (Green 1995) extends MH to handle that by including a Jacobian term to keep the move reversible. In GenomegaMap, block boundaries are proposed via a geometric distribution and split/merge moves are tossed in alongside the ordinary updates. You don't need the math today; just know that's what's happening when you see "RJ‑MCMC" in the source.

6. GenomegaMap proper

Why this section matters

All the pieces above — codon model, Bayesian likelihood, MCMC — were already present in Wilson's 2006 omegaMap. So what makes GenomegaMap different, and why does it scale to 10,000+ isolates? Three engineering wins, summarized here.

In plain English omegaMap (2006) was beautiful but slow — its likelihood took longer to compute as more isolates were added, so 10,000 genomes was hopeless. GenomegaMap (2020) replaces the slow likelihood with a tractable closed‑form expression that doesn't care how many isolates you have. Same Bayesian guts, ~100× faster on big data.

Wilson's original 2006 omegaMap already did Bayesian dN/dS via MCMC, but it scaled quadratically with sample size because it relied on the Li‑Stephens "product of approximate conditionals" (PAC) likelihood. With ~10,000 isolates per gene becoming routine in TB surveillance, that's intractable. GenomegaMap (Wilson 2020) makes three key changes:

1

Constant time per site

Replaces PAC with a population‑genetics derivation that gives a closed‑form Multinomial‑Dirichlet at each codon. Computational cost doesn't grow with sample size.

2

Phylogeny‑free

Doesn't reconstruct a tree. It takes raw allele counts and uses a diffusion‑coalescent integral over implicit genealogies. This is huge: phylogenetic methods give false positives in the presence of recombination.

3

Parent‑dependent mutation

Handles general mutation models where rates depend on both the source and target codon. The "parent‑dependent" extension gives a Dirichlet on allele frequencies conditional on the oldest allele.

Why phylogeny‑free is a big deal

Pathogen genomes are mostly clonal but recombine occasionally. Phylogenetic methods assume a single tree at every site, so a recombination event creates a phylogenetic inconsistency that the model can only "explain" by inferring fast mutation — i.e., a false positive signal of selection. By integrating over genealogies, GenomegaMap sidesteps this. The price: it assumes independence between sites, which isn't strictly true under linkage. Wilson's coalescent simulations show this is forgivable in practice; even fully clonal data give 92% coverage of true ω in 95% credible intervals.

The diversity‑divergence equilibrium assumption
Under Wright‑Fisher with weak selection at low mutation rates, the expected allele‑frequency distribution is approximately Dirichlet with concentration parameters proportional to the mutation rate matrix. GenomegaMap leverages this equilibrium so that, given Q (which contains ω), the predicted allele counts at any codon follow a tractable distribution. No tree needed.

7. Three model variants — Constant, Independent, Window

Why this section matters

This is the central pedagogical insight of the Ioerger & Shatby 2025 paper: "selection scan" isn't a single thing — it's a resolution choice. Pick the wrong resolution and you'll either smear out a real signal or drown in false positives. This section is the core skill of GenomegaMap.

In plain English You can ask "is this gene under selection?" at three zoom levels. Whole gene: one number per gene — great for ranking thousands of genes, but a single hot codon gets averaged away. Per codon: one number per codon — pinpoints exact mutations, but each number is built from very little data so it's noisy. Sliding window: the algorithm picks the right zoom for each region automatically — best of both, but heavier to run.
For today's project
Start with the constant model. That gives one ω posterior per gene per cohort, which is exactly what you need for the first diabetes vs non-diabetes comparison. The codon and window models are useful later when you want to localize a signal inside a gene.

The same GenomegaMap inference machinery can be run at three different scales, and the choice matters enormously for the questions you can answer:

ModelWhat ω doesStrengthWeakness
Constant (gene‑level) One ω for the whole gene High statistical power; clean credible interval per gene Single positively‑selected codon gets averaged out by the rest of the gene
Independent (codon‑level) One ω per codon, no sharing Pinpoints exact mutation hot‑spots Very low information per codon → high false‑positive rate. Long genes spuriously light up.
Sliding window One ω per contiguous block; block boundaries are unknown and inferred via RJ‑MCMC Best of both worlds — adapts the resolution to the data More computationally demanding; mean window size is itself a quantity

Why this section matters

You'll see both names in the literature. Knowing the difference (and the year they came out) is enough to navigate any methods discussion.

LikelihoodLi‑Stephens PAC approximationDiffusion‑coalescent Multinomial‑Dirichlet

Day to day in the lab, you will mostly invoke genomegaMap — but you will see the older paper cited and you'll want to be able to explain the difference in a Slack thread.

9. Running GenomegaMap in practice

Why this section matters

All the theory is great, but you need to actually run the tool. This is the five‑step recipe for going from a multiple sequence alignment to a posterior on ω.

GenomegaMap is bundled in Daniel Wilson's genomegaMap repo and ships with the gcat framework. Typical workflow:

  1. Align CDSs. Multiple‑sequence‑align the coding sequence of one gene across all isolates (use the H37Rv codon coordinates as the reference frame in TB work). Mask gaps and premature stops.
  2. Convert to input format. Codon counts per site, packaged as a single XML file per gene (the format gcat expects).
  3. Run MCMC. ~100k iterations, sampling every ~250 (so you save ~400 samples after thinning).
  4. Discard burn‑in. Drop the first 20% of saved samples — the chain hadn't reached stationarity yet.
  5. Summarize the posterior. Read out posterior mean ω and its 95% credible interval. Flag genes with the entire CI > 1 as positively selected.
# post‑MCMC analysis: read sampled omega values for one gene
import numpy as np, pandas as pd

# GenomegaMap writes one row per saved iteration; column 'mu_codon.omega' is ω
samples = pd.read_csv("rpoB.gcat.out", sep="\t")
omega   = samples["mu_codon.omega"].values

# Burn‑in: drop first 20% of post‑warmup samples
omega = omega[len(omega)//5:]

mean_omega = omega.mean()
lo, hi     = np.percentile(omega, [2.5, 97.5])

# "Significantly under positive selection" if entire 95% CI > 1
under_pos_sel = lo > 1.0

print(f"omega = {mean_omega:.2f}  CI=[{lo:.2f}, {hi:.2f}]  pos‑sel? {under_pos_sel}")

# Run gcat on a single gene's codon alignment
gcat \
  --params  params.xml  \
  --data    rpoB_codons.txt  \
  --out     rpoB.gcat.out  \
  --niter   100000  \
  --thin    250  \
  --seed    42

# For a 4000‑gene M. tuberculosis genome you'll want a job array on a cluster
sbatch --array=1‑3979 run_gene.slurm

# Compare two cohorts (e.g., diabetic vs non‑diabetic) for one gene
import numpy as np

omega_d  = load_samples("rpoB_diabetes.out",  burn=0.2)
omega_nd = load_samples("rpoB_nodiab.out",    burn=0.2)

# GenomegaMap doesn't natively compare cohorts. Bootstrap the difference distribution:
B = 10000
diffs = np.random.choice(omega_d, B) - np.random.choice(omega_nd, B)

mean_diff = diffs.mean()
lo, hi    = np.percentile(diffs, [2.5, 97.5])
p_d_gt_nd = (diffs > 0).mean()

print(f"Δω = {mean_diff:.2f}  95% CI [{lo:.2f}, {hi:.2f}]  P(d>nd)={p_d_gt_nd:.3f}")
Watch out
GenomegaMap analyzes one population at a time. The two‑cohort comparison shown above is a post‑hoc bootstrap over independently‑run chains — it's a reasonable approximation but not as principled as a joint hierarchical model. The Shatby thesis is upfront about this limitation.
